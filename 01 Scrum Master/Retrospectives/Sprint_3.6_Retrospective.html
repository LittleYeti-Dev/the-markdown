⚡ NEW ARCHITECTURAL PATTERN

DOM Enrichment via Client-Side JS Micro-Snippet

Sprint 3.6 introduced the first client-side JavaScript in the micro-snippet stack. Snippet #120 enriches the DOM at runtime by reading existing text content and injecting data attributes — enabling CSS-only targeting of dynamically-generated WordPress content without modifying PHP templates.

This pattern decouples visual styling from server-side rendering. CSS selectors can target attributes that don't exist in the raw HTML, because the JS bridge adds them before paint.

// Pattern: Read DOM text → Map → Inject attribute → CSS targets attribute document.querySelectorAll('.md-block').forEach(function(b) { var d = b.querySelector('.md-block-domain'); if (!d) return; var key = map[d.textContent.trim().toLowerCase()]; if (key) b.setAttribute('data-domain', key); }); // CSS cascade via custom property .md-block[data-domain="ai"] { --block-accent: var(--accent-ai); } .md-block[data-domain="cyber"]{ --block-accent: var(--accent-cyber); }
🔍 RETROSPECTIVE ANALYSIS

✅ WHAT WORKED

  • Micro-snippet architecture (DR-0016) — 4 separate visual problems solved by surgical snippet updates. No monolithic rewrites. Code Snippets REST API deployed all 4 without touching GitLab or WP admin.
  • CSS Custom Properties cascade — --block-accent variable set once per domain, inherited by all child elements. Scales cleanly for new domains.
  • JS DOM enrichment pattern — Solved the data-domain gap without refactoring S1.14 PHP. Opens a door for future client-side enrichment.
  • Zero-error deployment — 4/4 API calls returned 200 on first attempt. No rollbacks, no conflicts, no credential issues.
  • Full traceability — GitLab milestone, 5 issues, local artifacts, snippet inventory, DevOps folder — all synced in one session.

❌ WHAT DIDN'T

  • Per-category color gap not caught pre-deploy — CSS selectors targeted [data-domain] attributes that didn't exist in the DOM. Required a diagnostic cycle + new snippet (#120) to fix. Should have inspected DOM structure before writing CSS.
  • Two rendering stacks diverging — S1.14 shortcode (ns_frontpage) and Sprint 3.5+ static HTML (.md-block) are separate rendering paths for the same page. Neither is aware of the other. This is technical debt.
  • No visual diff process — Prototype-to-production comparison was manual eyeballing. No screenshot baseline, no automated visual regression.

⚠️ WHAT CHANGES

  • DOM audit before CSS deploys — New standing order: verify target elements have expected attributes/classes in live DOM before deploying CSS that depends on them.
  • Resolve dual rendering stack (S3.7) — Architectural decision required: S1.14 shortcode pipeline vs Sprint 3.5+ static stack. Blocks all Phase B work in S3.8.
  • Screenshot capture in deploy flow — Before/after screenshots for every snippet deploy. Manual is fine. Stored in DevOps folder.
