## 2. Handling Rules

### Universal Rules (Apply to ALL Credentials)

1. **Credentials are provided by Yeti at session start — ASK IMMEDIATELY.** Do not wait until you need them. Request all credentials upfront so you can work efficiently.

2. **In-memory only.** NEVER write any credential to:
   - Any file on disk (`.md`, `.txt`, `.json`, `.env`, `.php`, etc.)
   - Any git commit message or commit body
   - Any log file or output document
   - Any code comment or inline documentation
   - Any chat message that could be logged

3. **NEVER include in outputs.** If generating a document, report, or code that references credentials, use placeholder text like `[GITLAB_PAT]` or `[APP_PASSWORD]` — never the actual value.

4. **Assume short-lived.** Any credential provided in-session may be rotated within hours. If a credential stops working, do not troubleshoot — ask Yeti for a fresh one.

5. **No credential caching.** Do not store credentials in variables that persist beyond the immediate API call. Use them, then discard them from working memory.

6. **Session isolation.** Credentials from one session do not carry to the next. Every new session starts with zero credentials and must request them fresh.

## 4. Security Boundaries

### Root Secret: Vault Encryption Key

The vault encryption key (Snippet ID 31, priority 1) is the root secret for the entire credential infrastructure.

| Property | Value |
|----------|-------|
| Location | Code Snippet ID 31 — `NS Vault Key` |
| What it protects | All values stored in Token Vault (S1.17) — currently Claude API key |
| Encryption | AES-256-CBC |
| Known trade-off | Key is visible in the Code Snippets editor (DR-0012) — mitigated by admin-only access |
| If compromised | Rotate ALL vault contents immediately using Key Rotation procedure (S1.20) |

### Key Rotation Procedure (S1.20)

If the vault key is suspected compromised:

1. Generate a new encryption key
2. Call `ns_vault_rotate_key()` — this decrypts all values with the old key and re-encrypts with the new key
3. Update Snippet ID 31 with the new key constant
4. Verify all vault retrievals still work
5. Log the rotation event via `ns_audit_log()`

### Audit Trail

All vault operations are captured by:
- **Audit Logging (S1.22):** Admin action trail — tracks who accessed what
- **Diagnostic Logger (S1.23):** System event capture — tracks errors and warnings
- Both logs are accessible via REST API (`ns/v1/diagnostics/health`) and admin pages

### What Must NEVER Be Stored

The following must NEVER appear in any file, document, or artifact in this project:

- GitLab PAT values
- WordPress Application Password values
- Claude API key values
- Any OAuth tokens or session cookies
- Any API keys from social platforms (Sprint 2+)
- Any encryption keys (except in their designated Code Snippet)

## 5. Sprint 2+ Credential Planning

Sprint 2 adds 5 social platform API integrations. Each will require OAuth credentials:

| Platform | Expected Credential Type | OAuth Flow | Storage Plan |
|----------|------------------------|------------|--------------|
| X (Twitter) | OAuth 2.0 PKCE tokens | Authorization Code + PKCE | Token Vault (S1.17) |
| LinkedIn | OAuth 2.0 tokens | Authorization Code | Token Vault (S1.17) |
| Instagram | Graph API token via Facebook | Facebook OAuth | Token Vault (S1.17) — blocked by BLK-001 |
| YouTube | API key + OAuth 2.0 | Authorization Code | Token Vault (S1.17) |
| Medium | Bearer token | Simple token auth | Token Vault (S1.17) |

All Sprint 2 OAuth tokens will be stored in the Token Vault using the same AES-256-CBC encryption. Token refresh automation (S2.7) will handle expiration. Platform API error handling (S2.8) will ensure no tokens leak in error responses.

This credential protocol extends naturally to Sprint 2 — no architectural changes needed.
